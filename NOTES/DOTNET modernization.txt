Good instinct to check. Mainframe modernization is a niche; .NET modernization is a much larger adjacent market where your profile is arguably stronger, and most of the same transferable skills apply. Here's the honest map:

What the work actually is
Large enterprises have tens of millions of lines of .NET Framework code (often 15-25 years old) that they need to get off aging Windows Server data centers. The typical migration patterns:

.NET Framework 4.x → .NET 8/9 (cross-platform, Linux-friendly)
WCF → gRPC or ASP.NET Core Web API / REST
WebForms / WinForms / WPF → Blazor, React, or Angular SPAs
IIS on Windows Server → Kestrel in containers on Linux, typically Kubernetes
SQL Server on-prem → Azure SQL / AWS RDS / PostgreSQL (the last one for license-cost reasons)
Forms Auth / WS-Fed / WIF → OAuth 2.0 / OIDC (you literally did this at Akruto for Microsoft 365)
Monolith → microservices (you did this at Cimpress)
Why there's real budget: Windows Server 2012/2016 end-of-support cycles, board-level cloud mandates, SQL Server / Windows licensing cost pressure, and the inability to hire new grads who know WebForms or WCF.
